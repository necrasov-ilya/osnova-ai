# OSNOVA AI Project

## Суть проекта

OSNOVA AI - мультимодальное Android-приложение для live-генерации конспектов из визуального потока камеры.

Пользователь наводит камеру на доску, слайд, тетрадь, экран или учебный материал. Приложение в реальном времени понимает, что попало в кадр, выбирает маршрут обработки через OCR, VLM или LLM-пайплайн, собирает смысловой черновик и постепенно дополняет текущую заметку структурированным конспектом.

Продукт должен ощущаться как умный редактор заметок. Камера не живёт отдельно от текста. Визуальная очередь показывает этапы обработки, готовые фрагменты появляются в конспекте по мере генерации, каждый добавленный блок можно проверить, принять, отредактировать или удалить.

## Черновик пайплайна

Держи этот пайплайн как рабочий черновик. Его можно уточнять по мере развития архитектуры, прототипов и документации.

0. Camera Activation
   Камера и анализ запускаются только когда пользователь раскрывает камерный контейнер в заметке. В закрытом состоянии приложение не анализирует кадры в фоне.

1. Live Camera Stream
   Получить непрерывный поток кадров с камеры.

2. Frame Sampler
   Брать кадр раз в 300-700 мс. Уменьшать разрешение для быстрой первичной проверки.

3. Text / Scene Router
   Определить тип материала: печатный текст, почерк, доска, слайд, формула, схема, экран, пустой кадр или смешанный материал.

4. Stability Gate
   Проверить, что кадр достаточно резкий, текст стабилен 2-3 срабатывания подряд, регион не прыгает, а контент отличается от уже обработанного.

5. Processing Queue
   Создать визуальную карточку задачи: вижу текст, распознаю, собираю черновик, вставляю в конспект.

6. Route Selection
   Выбрать маршрут обработки:
   - печатный текст -> OCR
   - почерк -> VLM
   - формулы -> VLM / LaTeX pipeline
   - схемы и слайды -> OCR + VLM
   - смешанный кадр -> разбиение на регионы и комбинированная обработка

7. Content Extraction
   Получить сырой текст, формулы, структуру слайда, связи на схеме или описание визуального материала.

8. Draft Builder
   Очистить результат, убрать мусор, восстановить структуру и превратить материал в черновик конспекта.

9. Note State Engine
   Учитывать текущую заметку: активный раздел, последние абзацы, стиль, тему и место, на котором пользователь остановился.

10. Smart Insert
    Решить, куда добавить фрагмент: продолжить абзац, создать подраздел, добавить список, таблицу, формулу или пояснение.

11. Final Note Block
    Показать готовый блок в конспекте. Схлопнуть карточку задачи и убрать её из очереди.

Текущие состояния live-задачи: `Sampling`, `StableRegion`, `RawDraft`, `GeneratingMarkdown`, `Inserted`, `Failed`.

Gemma в Android MVP должна отвечать потоково через LiteRT-LM `Conversation.sendMessageAsync(...)`. UI принимает события `start`, `raw_draft`, `delta`, `done`, `error`. По `delta` обновляй provisional Markdown-блок, чтобы пользователь видел живую генерацию, а не пустое ожидание.

Заметка хранится как список `NoteBlock`, а не как одно большое поле. Каждый блок хранит `id`, `markdown`, `rawSource`, `status` и timestamps. Markdown рендерится нативно через Markwon, редактирование открывает исходный Markdown.

## Производительность

Держи pipeline событийным. Дешёвые проверки могут работать часто, тяжёлые VLM/LLM-маршруты запускаются только после stability gate.

Предусматривай режимы: энергосбережение, сбалансированно, производительность. Режим меняет частоту sampler, разрешение анализа, агрессивность gate и готовность запускать VLM.

На этапе продуктового формирования режимы производительности не являются отдельной фичей. Не экономь лишний раз в ущерб проверке главного сценария. Архитектуру готовь так, чтобы позже можно было менять pipeline по режимам без переписывания ядра.

## Модель И Рантайм

Основной on-device runtime для Android: LiteRT-LM. Не предлагай Ollama, LM Studio, Termux-серверы или случайные GGUF-обёртки как основной путь приложения.

Главный модельный кандидат: Gemma E2B в формате `.litertlm`. Для рабочего Android MVP использовать generic `gemma-4-E2B-it.litertlm` через GPU/CPU. Qualcomm SM8750 артефакт Gemma4-E2B держать как NPU-кандидат для Vivo X200 Ultra, но не как блокер MVP, пока NPU-ветка не прошла smoke-тест. Fallback для исследования - Gemma-3n-E2B. Bonsai держать как text-only исследование, но не как основной VLM для камеры.

Gemma4-E2B не должна быть блокером для разработки. Если она тормозит продуктовый прототип, используй серверный fallback, мок-ответы или подготовленные тестовые изображения, пока UI, очередь, Note State Engine и Smart Insert продолжают развиваться.

В Android MVP Gemma запускается через LiteRT-LM в отдельном процессе `:gemma`. Ошибка или падение модельного слоя не должны закрывать экран заметки. OCR-результат остаётся запасным материалом для вставки.

Prompt Gemma для MVP: вернуть только Markdown, без обрамления, не выдумывать, сохранить формулы и списки, явно помечать неразборчивые места.

Русский OCR не доверяй латинскому ML Kit. Для кириллицы в MVP используется Tesseract `rus+eng` как локальный fallback, а перед вставкой обязательно работает quality gate / DraftBuilder. Сырой OCR-шум не должен попадать в заметку напрямую.

Первое целевое устройство для тестов: Vivo X200 Ultra на Snapdragon 8 Elite.

Модель не должна ехать внутри каждого APK. Храни `.litertlm` как отдельный версионированный артефакт с manifest, `sha256` и локальным кэшем. Обычные debug-обновления приложения не должны заново скачивать модель. Для разработки с девайсом допускай debug override через app-specific external storage.

## Документация

Вести документацию на русском языке. Основной стиль - ясный, местами неформальный, без лишней торжественности.

Пиши так, будто объясняешь проект внимательному разработчику, который хочет быстро войти в контекст. Не используй пустые нейронные фразы вроде "важно отметить", "стоит подчеркнуть", "данный подход позволяет" без реальной пользы. Не строй текст на постоянных противопоставлениях. Не превращай каждый абзац в схему с двоеточием или точкой с запятой.

Для проектной документации mdBook считается подходящим базовым форматом. Когда документация будет заведена, держи её в предсказуемой структуре:

- `docs/book.toml`
- `docs/src/SUMMARY.md`
- `docs/src/*.md`

Начинай с материалов, которые помогают принимать решения: обзор продукта, пользовательский сценарий, пайплайн, архитектура, заметка как состояние, UI-поведение, дизайн-система, ограничения, открытые вопросы.

Не раздувай документы заранее. Лучше короткая страница, которую легко дописать, чем длинный текст, который страшно трогать.

## Дизайн-Система

Всё, что связано с палитрой, типографикой, отступами, формами, состояниями, компонентами, анимациями и визуальным языком, веди как полноценную дизайн-систему OSNOVA AI.

Material можно и нужно использовать как базу для Android-паттернов, но продукт не должен быть безликим Material-приложением. Компоненты проекта могут быть собственными или смешанными с Material, если это помогает UX заметок, камеры и очереди обработки.

Базовая палитра MVP:
- акцент - `#AC2954`
- дневной фон - `#F5F2EB`
- тёмная тема - `#141413`

Акцент общий для обеих тем. Системные панели Android должны поддерживать активную тему, а не выбиваться отдельным цветом.

Фиксируй решения по дизайну как системные правила, а не как разовые правки экрана. Если меняется палитра, кнопка, карточка очереди, состояние блока конспекта или стиль редактора, обновляй документацию дизайн-системы.

Markdown-блок, raw draft card, processing queue и форматирующая пилюля edit sheet считаются компонентами дизайн-системы. Raw draft показывается только как временный источник для Gemma и визуально сжимается при генерации. В заметке остаётся Markdown-блок со статусом `generating`, `ready`, `edited` или `failed`.

## Коммиты

Пиши коммиты на русском языке, но сохраняй понятную conventional-маску:

```text
feat(scope):
- Добавлен ...
- Реализован ...
- Изменён ...
```

Используй формы результата, а не инфинитив:

- Сделан
- Реализован
- Добавлен
- Изменён
- Исправлен
- Обновлён
- Пересобран
- Удалён
- Подключён
- Описан
- Настроен

Не используй формы вроде "Собрать", "Добавить", "Исправить", "Пересобрать" в пунктах тела коммита.

Типы коммитов держи короткими и привычными:

- `feat` - новая функция или заметная возможность
- `fix` - исправление поведения
- `docs` - документация
- `style` - визуальные изменения без смены логики
- `refactor` - переработка кода без нового поведения
- `test` - тесты
- `chore` - обслуживание проекта
- `build` - сборка, зависимости, конфиги
- `ci` - автоматизация и проверки

Scope выбирай по смыслу: `docs`, `pipeline`, `camera`, `notes`, `editor`, `design-system`, `android`, `ocr`, `vlm`, `llm`, `queue`.

Пример:

```text
docs(pipeline):
- Описан черновой live camera pipeline
- Добавлены правила маршрутизации OCR, VLM и LLM
- Зафиксированы этапы очереди обработки
```

Если изменение маленькое, достаточно одного-двух пунктов. Если изменение большое, добавляй столько пунктов, сколько нужно, чтобы через месяц было понятно, что реально изменилось. Не превращай тело коммита в пересказ каждого файла.

## Рабочее Правило

Когда задача касается OSNOVA AI, сначала сверяйся с этой проектной памятью. Если идея меняется, обновляй skill вместе с документацией, чтобы будущие обсуждения не начинались с нуля.
