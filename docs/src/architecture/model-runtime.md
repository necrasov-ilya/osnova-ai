# Модель и рантайм

Нам нужна модель не для чата ради чата. Ей надо смотреть на стабильный визуальный фрагмент, описывать учебный материал и помогать вставлять блок в заметку.

## Решение на сейчас

Основной on-device путь: **Gemma E2B через LiteRT-LM**.

Если брать текущую линейку Google AI Edge, главный кандидат для OSNOVA AI - `Gemma4-E2B` в формате `.litertlm`. Если на целевых устройствах она окажется слишком тяжёлой или нестабильной, откатываемся на `Gemma-3n-E2B`.

В текущем MVP используется `com.google.ai.edge.litertlm:litertlm-android:0.12.0` и файл `gemma-4-E2B-it.litertlm`. Это обычный Android-артефакт Gemma 4 E2B для запуска через GPU или CPU.

Генерация идёт потоково через `Conversation.sendMessageAsync(...)`. UI получает не один большой ответ, а цепочку `start`, `raw_draft`, `delta`, `done` или `error`. Это нужно для главного ощущения продукта: человек видит, как заметка пишется прямо во время пары.

Файл `gemma-4-E2B-it_qualcomm_sm8750.litertlm` остаётся в исследовательской ветке для NPU на Snapdragon 8 Elite / SM8750. Мы не делаем его блокером MVP: если NPU-ветка зависает или не поднимается, приложение должно продолжать работать на generic Gemma и OCR fallback.

Gemma4-E2B не должна быть блокером для продуктовой разработки. Если модель долго запускается, плохо держится в памяти или временно не проходит на устройстве, продолжаем строить UX, очередь, Note State Engine и Smart Insert через fallback: серверный VLM/LLM, мок-ответы или заранее подготовленные тестовые изображения.

Почему так:

- модель мультимодальная;
- есть официальный Android-путь;
- LiteRT-LM даёт Kotlin API;
- поддерживаются CPU/GPU/NPU backends;
- есть Google AI Edge Gallery как живой пример Android-приложения;
- можно стримить ответ через coroutine `Flow`;
- можно отдельно задавать backend для vision/audio;
- это не серверный раннер и не обвязка вокруг десктопных локальных LLM.

## Рантайм

Берём **LiteRT-LM** как основной runtime для on-device LLM/VLM.

Не берём для Android-приложения:

- Ollama;
- LM Studio;
- Termux-сценарии;
- самодельные серверы внутри телефона;
- случайные GGUF-обёртки без нормального Android lifecycle.

LiteRT-LM хорош именно тем, что он нативно ложится в Android-разработку: Gradle dependency, Kotlin API, `Engine`, `Conversation`, backend selection, streaming, мультимодальные `Content`.

## Модельные роли

| Роль | Модель | Где работает | Комментарий |
| --- | --- | --- | --- |
| Основной MVP-слой | Gemma4-E2B `.litertlm` | on-device GPU/CPU | Собирает структурный блок по OCR-тексту. Это рабочая ветка для первого MVP. |
| NPU-кандидат | Gemma4-E2B SM8750 `.litertlm` | Snapdragon 8 Elite / NPU | Исследуем отдельно. Не должен держать UX и редактор заложниками. |
| Консервативный VLM | Gemma-3n-E2B `.litertlm` | on-device | Более понятный edge-кандидат из линейки 3n. Хорош для эксперимента и fallback. |
| Лёгкий текстовый слой | Gemma3-1B или Qwen2.5-1.5B | on-device | Может собирать короткие черновики по OCR-тексту, когда vision не нужен. |
| Tool слой | FunctionGemma | on-device | Интересно позже для структурных действий и function calling. Для MVP не основной путь. |
| Bonsai | пока не основной | исследование | Интересная плотная text-only ветка, но не закрывает камеру и мультимодальность сама по себе. |

## Как запускать

Черновая схема Android-интеграции:

1. Модель хранится отдельным загружаемым файлом, а не зашивается в базовый APK.
2. При первом включении on-device режима приложение предлагает скачать модель по Wi-Fi.
3. `ModelManager` проверяет доступность модели, свободное место и режим питания.
4. `InferenceEngine` поднимает LiteRT-LM `Engine` в фоне.
5. Для текущего MVP отправляем очищенный OCR draft и короткий prompt.
6. Ответ стримится в Markdown-блок заметки.
7. Для будущего VLM-запроса отправляем `ImageFile` или `ImageBytes` плюс короткий prompt.
8. Note State Engine получает не сырой OCR, а готовый Markdown или fallback после clean-up.

Важно: инициализация модели может занимать секунды. Её нельзя делать на UI thread и нельзя запускать каждый раз при новом кадре.

Первое реальное устройство для проверки on-device пути: Vivo X200 Ultra на Snapdragon 8 Elite. На нём проверяем старт модели, скорость ответа, нагрев и поведение backend.

Подробный план хранения, обновления и debug-доставки модели описан в [Доставке моделей](model-delivery.md).

## Как использовать в пайплайне

VLM запускается только после `Stability Gate`.

Пример события:

```text
StableVisualRegion(
  type = "board_handwriting",
  image = cropped_frame,
  noteContext = last_blocks,
  task = "extract_and_explain"
)
```

Prompt для VLM должен быть коротким и прикладным:

```text
Опиши учебный материал на изображении. Вытащи текст, формулы и связи.
Не фантазируй. Если часть не читается, пометь это явно.
Верни результат как черновик конспекта.
```

Prompt для текущей Gemma-ветки:

````text
Преврати распознанный учебный материал в короткий Markdown-конспект.
Верни только Markdown без вступлений и без ```markdown.
Не выдумывай факты. Если место не читается, напиши "неразборчиво".
Сохраняй списки, формулы и связи, если они есть.
Пиши по-русски.
````

Для Smart Insert лучше отдельный шаг:

```text
Вот текущая заметка и новый черновик. Реши, куда его вставить:
продолжить абзац, добавить пункт, создать подраздел, добавить формулу
или оставить блок на проверку.
```

Так мы не смешиваем распознавание изображения и редактирование заметки в один тяжёлый запрос.

## Режимы

Энергосбережение:

- не держать VLM постоянно загруженной, если это бьёт по памяти;
- использовать OCR + лёгкий текстовый слой;
- VLM запускать вручную или только по сильному сигналу.

Сбалансированно:

- держать модель прогретой во время активной камеры, если устройство тянет;
- запускать VLM по стабильным событиям;
- ограничить параллельность одной тяжёлой задачей.

Производительность:

- GPU/NPU backend, если он стабилен на устройстве;
- speculative decoding/MTP, если модель и backend поддерживают;
- более частые VLM-события, но всё равно не на каждый кадр.

## Что надо проверить руками

- Vivo X200 Ultra: baseline для первых on-device замеров.
- Сколько места занимает модель после загрузки и кэша.
- Сколько времени занимает первый запуск.
- Как ведёт себя GPU на Snapdragon, Tensor и MediaTek.
- Есть ли нормальный CPU fallback.
- Что происходит при закрытии камеры во время генерации.
- Насколько хорошо модель читает доску, слайды, экран и почерк.
- Сколько правок пользователь делает после вставки блока.
