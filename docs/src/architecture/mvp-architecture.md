# MVP-архитектура приложения

Этот MVP должен открываться на телефоне, включать камеру только по действию пользователя, распознавать стабильный текстовый регион и постепенно собирать Markdown-блок в заметке. Без идеальной VLM-магии, но уже с ощущением live-конспекта, а не разовой OCR-вставки.

## Что входит в первый MVP

- Kotlin Android-приложение.
- Главный экран заметок.
- Создание заметки.
- Экран заметки с нижней пилюлей.
- Раскрывающийся камерный контейнер.
- Проверка и скачивание OCR-модуля при первом запуске.
- Проверка и скачивание Gemma 4 E2B `.litertlm`.
- CameraX preview.
- ML Kit Text Recognition как быстрый первый проход.
- Tesseract `rus+eng` как локальный fallback для кириллицы.
- LiteRT-LM runtime dependency.
- Запуск Gemma через LiteRT-LM в отдельном процессе `:gemma`.
- Потоковый ответ Gemma через `sendMessageAsync(...)`.
- Очередь обработки.
- Raw draft card в камерном слое.
- Markdown-блоки заметки с состояниями генерации.
- Markwon preview для готового Markdown.
- Две темы с одинаковым акцентом и разным фоном.

## Что пока не входит

- Полноценный VLM-анализ изображения без OCR-промежуточного слоя.
- Надёжная настройка всех backend LiteRT-LM под разные SoC.
- Умный LLM Smart Insert по структуре всей заметки.
- Режимы производительности как пользовательский тумблер.

Gemma в MVP обязательна, но её текущий слой честно допускает runtime-ошибку. Если LiteRT-LM на конкретной сборке не поднимется, OCR-поток всё равно отдаст очищенный черновик, а очередь покажет состояние Gemma.

Gemma запускается не в основном процессе интерфейса, а в сервисе `GemmaWorkerService`. Это нужно для разработки на реальном устройстве: если нативный runtime модели упадёт или не поднимет backend, экран заметки должен продолжить жить и сохранить OCR-результат.

Сырой OCR не вставляется в заметку напрямую. Сначала работает quality gate и `DraftBuilder`: убирает шумовые строки, ждёт стабильного повтора и только потом отдаёт чистый черновик в Gemma. Если Gemma не поднялась, очищенный черновик используется как fallback-блок.

## Live states

Задача обработки идёт через явные состояния:

| Состояние | Что происходит | Что видит пользователь |
| --- | --- | --- |
| `Sampling` | Камера отдаёт кадры, дешёвый gate проверяет резкость, яркость и шум. | Плашка `ищу текст`, очередь может показывать `держу кадр`. |
| `StableRegion` | Текстовый регион повторился несколько раз и не скачет. | В очереди появляется `Регион`. |
| `RawDraft` | ML Kit и Tesseract дали сырой материал, `DraftBuilder` его чистит. | Появляется raw draft card. |
| `GeneratingMarkdown` | Gemma получает черновик и стримит Markdown. | Блок в заметке пишет себя постепенно, raw card визуально сжимается. |
| `Inserted` | Поток закончен, блок стал готовым. | Очередь схлопывает задачу, блок получает статус `готово`. |
| `Failed` | Gemma не стартовала, зависла или вернула ошибку. | Вставляется fallback-блок, ошибка остаётся в очереди без падения экрана. |

Камерный контейнер можно закрыть во время генерации. Это закрывает preview и analyzer, но не отменяет уже запущенную задачу Gemma. Блок продолжает дописываться в заметке.

## Первый запуск

При старте приложение проверяет доступность OCR-модуля ML Kit Text Recognition через Google Play Services и наличие файла Gemma.

Если чего-то нет, пользователь видит короткий экран с кнопкой скачивания. Кнопка запускает официальный `ModuleInstallClient` для OCR, скачивает `rus/eng.traineddata` для Tesseract и скачивает Gemma 4 E2B с Hugging Face.

Модель хранится отдельно от APK во внутреннем хранилище приложения и не скачивается заново при обычном `installDebug`.

## Runtime-поток

```text
MainActivity
↓
ModelGate
↓
NotesHome
↓
NoteEditor
↓
CameraPanel
↓
TextRecognitionAnalyzer
↓
DraftBuilder / Quality Gate
↓
GemmaWorkerService (:gemma)
↓
GemmaRuntime
↓
Stream events
↓
ProcessingQueue
↓
NoteBlock list
```

## Stream events Gemma

`GemmaWorkerService` больше не отправляет один финальный результат. Он шлёт последовательность событий:

| Событие | Смысл |
| --- | --- |
| `start` | Модельная задача принята. |
| `raw_draft` | UI получил очищенный источник, можно показать raw card. |
| `delta` | Новый кусок Markdown, который добавляется к generating-блоку. |
| `done` | Финальный Markdown готов, блок становится `ready`. |
| `error` | Runtime не справился, блок получает fallback. |

Так мы получаем поведение как в FastVLM-демо: пользователь видит не пустое ожидание, а живой блок, который растёт по мере ответа модели.

## Smart Insert в MVP

Сейчас Smart Insert простой:

- если заметка пустая, создаётся первый `NoteBlock`;
- если есть активные блоки, новый блок добавляется в конец;
- повторы отсекаются по похожести недавнего текста;
- `generating`-блок обновляется по stream delta;
- готовый блок можно открыть и править как исходный Markdown.

Позже этот слой станет отдельным Note State Engine с LLM-подсказкой и структурной вставкой.
