# Режимы производительности

В будущем нужен тумблер режима работы:

- энергосбережение;
- сбалансированно;
- производительность.

Это не косметическая настройка. Режим меняет частоту анализа, разрешение кадров, правила gate и доступность тяжёлых маршрутов.

## Статус

На этапе продуктового формирования режимы производительности не делаем как отдельную пользовательскую фичу. Сейчас важнее собрать главный сценарий: заметка, раскрытие камеры, очередь, распознавание, черновик и вставка в нужное место.

Архитектуру при этом готовим заранее. Pipeline должен иметь точки настройки: частота sampler, разрешение анализа, агрессивность gate, выбор OCR/VLM/LLM-маршрута и параллельность очереди. Когда дойдём до оптимизации, режимы будут менять эти параметры без переписывания всего приложения.

На первых этапах не экономим лишний раз. Если Gemma4-E2B помогает быстрее проверить продуктовый сценарий на Vivo X200 Ultra, используем её. Если модель мешает разработке, включаем fallback и продолжаем собирать продукт.

## Общее правило

Камера и анализ работают только при раскрытом камерном контейнере. Если пользователь закрыл камеру, новые кадры не анализируются.

Очередь задач может доработать уже принятые кадры, но pipeline не должен продолжать смотреть в мир через закрытую камеру.

## Энергосбережение

Для средних и слабых устройств, долгой пары, низкого заряда или нагрева.

Настройки:

- sampler: примерно 1 кадр в 900-1500 мс;
- preview остаётся плавным, но анализ редкий;
- разрешение анализа ниже;
- OCR запускается только после уверенного gate;
- VLM запускается вручную или по очень сильному сигналу;
- если on-device VLM слишком дорогая, используем OCR + лёгкую текстовую модель;
- LLM собирает блоки пачками, а не на каждый маленький фрагмент;
- при нагреве режим включается автоматически.

Ощущение для пользователя: чуть медленнее, зато телефон не превращается в грелку.

## Сбалансированно

Режим по умолчанию.

Настройки:

- sampler: примерно 1 кадр в 500-900 мс;
- ML Kit работает как дешёвый роутер;
- gate требует 2-3 стабильных срабатывания;
- OCR может работать часто, VLM только по событиям;
- Gemma E2B через LiteRT-LM держится готовой только если устройство это тянет;
- LLM вставляет блоки небольшими порциями.

Ощущение для пользователя: приложение живое, но без постоянной тяжёлой генерации.

## Производительность

Для сильных устройств, зарядки или коротких интенсивных сессий.

Настройки:

- sampler: примерно 1 кадр в 300-500 мс;
- выше разрешение для анализа;
- VLM запускается охотнее;
- GPU/NPU backend используется, если он стабилен на устройстве;
- можно чаще обновлять очередь и предварительные описания;
- больше параллельности, но с жёстким лимитом задач.

Ощущение для пользователя: быстрее появляются блоки, выше риск нагрева и расхода батареи.

## Автоматическое понижение

Приложение должно уметь само отступать:

- если Android сообщает о тепловом давлении;
- если задачи в очереди копятся быстрее, чем завершаются;
- если среднее время обработки растёт;
- если заряд низкий;
- если пользователь свернул экран или закрыл камеру.

Сначала уменьшаем частоту sampler. Затем понижаем разрешение. Потом отключаем автоматический VLM и оставляем только OCR/gate.

Для Android-сигналов смотрим на Thermal API. Не полагаемся на один статус как на абсолютную истину: на разных устройствах тепловые статусы и headroom могут вести себя по-разному.

## Метрики, которые надо собирать в прототипе

- время обработки кадра дешёвым слоем;
- время OCR;
- время VLM/LLM;
- количество отброшенных кадров;
- длина очереди задач;
- время до первого полезного блока;
- температура/thermal status, где доступно;
- заряд и режим питания;
- доля ручных правок пользователя после вставки блока.
