# Пользовательский поток

Это поток для первого большого сценария: начинается пара, пользователь открывает приложение и собирает конспект из того, что видит камера.

## Главный экран

Пользователь заходит в приложение и видит обычный интерфейс заметок:

- хедер
- нижнюю навигацию в пилюле
- карточки заметок по центру
- у карточки есть название и короткая сводка
- кнопка `+` создаёт новую заметку.

После нажатия `+` открывается шторка создания. Пользователь вводит название, подтверждает, затем приложение переводит его на экран заметки.

## Экран заметки

Заметка открывается как редактор. Камера не стартует сама. Это важное правило.

На экране есть:

- список Markdown-блоков заметки
- нижняя пилюля с действиями
- центральная кнопка, которая раскрывает камерный контейнер
- зона очереди обработки, когда камера активна
- raw draft card, если уже найден стабильный OCR-черновик

Камерный блок вдохновляется демо FastVLM: компактный контейнер, внутри которого видно изображение, статус обработки и результат. У нас этот блок должен быть частью редактора заметок, а не отдельной демо-страницей.

## Включение камеры

Пользователь нажимает центральную кнопку снизу. Камерный контейнер раскрывается через анимацию.

Только после раскрытия:

- поднимается camera preview;
- включается анализ кадров;
- запускается дешёвая проверка сцены;
- приложение начинает понимать, есть ли в кадре текст, доска, слайд, почерк, формула или другой учебный материал.

Если контейнер закрыт, приложение не должно распознавать кадры в фоне. Максимум - держать UI-состояние заметки.

## Работа во время пары

Пользователь наводит камеру на доску или экран. Приложение не отправляет каждый кадр в тяжёлую модель.

Рабочая схема:

1. Быстрый слой проверяет кадр.
2. Stability gate ждёт устойчивый материал.
3. Очередь показывает задачу.
4. Выбирается маршрут обработки.
5. Сырой результат превращается в очищенный draft.
6. Gemma стримит Markdown.
7. Generating-блок постепенно появляется в заметке.

Очередь должна быть видимой. Пользователь понимает, что приложение сейчас видит, что распознаёт и какой блок скоро появится.

Raw draft остаётся временным материалом. Он может сжиматься визуально, пока Markdown растёт в заметке, но сам сырой OCR не считается финальным текстом.

## Закрытие камеры

Пользователь закрывает камерный контейнер. Перед закрытием контейнер визуально сокращает накопленную информацию.

Это только анимация. Она не должна запускать новую генерацию сама по себе.

Генерация и вставка продолжаются в заметке. Если задача уже была поставлена в очередь, она может завершиться после закрытия камеры. Новые кадры после закрытия не анализируются.

## Вставка в заметку

Информация добавляется не просто в конец. Логика должна быть похожа на агентную работу с кодом: система смотрит на текущее состояние заметки и решает, где продолжить.

Возможные действия:

- продолжить текущий абзац;
- добавить пункт списка;
- создать подраздел;
- вставить формулу;
- добавить пояснение к уже написанному;
- создать блок, который ждёт подтверждения пользователя.

Каждый блок должен быть проверяемым: принять, отредактировать, удалить.

Для MVP редактирование открывает исходный Markdown в edit sheet. Форматирующая пилюля даёт быстрые действия: bold, italic, heading, list, quote, code.
